基本概念

租户 Tenant

每个租户都可以有单独的认证和授权机制，可以针对租户设置存储配额、消息生存时间TTL和隔离策略

命名空间

命名空间是租户的管理单元，每个租户下可以创建多个命名空间。可以通过在命名空间上设置的配置策略来管理该命名空间下的Topic，这样就可以在命名空间的级别上为该命名空间的所有Topic设置访问权限、调整复制设置、管理跨集群跨地域的消息复制，控制消息过期时间。

Topic

消息数据的载体，在 Pulsar 中 Topic 可以指定分为多个 partition，如果不设置默认只有一个 partition。

Topic命名URL定义：{persistent|non-persistent}://tenant/namespace/topic

  • persistent / non-persistent：表示主题的类型，主题分为持久化和非持久化主题，默认是持久化的类型。持久化的主题会将消息保存到磁盘上，而非持久化的主题就不会将消息保存到磁盘，只存在于内存中，当使用非持久topic分发时，关闭Broker或者关闭订阅者，非持久化Topic上所有的瞬时消息都会丢失，客户端可能会出现丢失消息的情况。 非持久topic中，broker会立即发布消息给所有连接的订阅者，而不会在BookKeeper中存储。 如果有一个订阅者断开连接，broker将无法重发这些瞬时消息，订阅者将永远无法收到这些消息了。
  • tenant：Pulsar 实例中主题的租户，租户对于 Pulsar 中的多租户至关重要，并且分布在集群中。
  • namespace：将相关联的 Topic 作为一个组来管理，是管理 Topic 的基本单元。每个租户可以有一个或多个命名空间。
  • topic：Pulsar 中的主题被命名为通道，用于将消息从生产者传输到消费者。

生产消息模式

路由模式:（和kafka非常像）

  • RoundRobinPartition: 默认策略，轮询方式将消息发布到所有分区
  • SinglePartition: 如果消息没有指定key，生产者将会随机选择一个分区，并发布所有消息到这个分区。如果消息指定了key，分区生产者会根据key的hash值将该消息分配到对应的分区，可以看源码
  • 自己实现路由方式

访问模式：

  • Shared: 默认值，多个生成者可以将消息发送到同一个Topic
  • Exclusive: 只有一个生产者可以将消息发送到Topic ，当其他生产者尝试发送消息到这个Topic时，会发生错误
  • WaitForExclusive: 只有一个生产者可以将消息发送到Topic。当已有生成者和Topic建立连接时，其他生产者的创建会被挂起而不会产生错误

订阅消息模式

https://huangzhongde.cn/post/Linux/Pulsar_Pub_Sub_Mode/

  • 独占模式(Exclusive)：一个订阅只能与一个消费者关联（同一个SubscriptionName只能由一个consume，多的会被拒绝），只有这个消费者可以接收到主题的全部消息，如果该消费者出现故障，消费过程将停止。（只能对收到的最后一条消息进行取消确认）

  • 共享模式(Shared)：消息通过轮询机制（或自定义机制）分发给不同的消费者，每个消息只会被分发给一个消费者，如果消费者断开连接，所有发送给它但未被确认的消息将被重新分配给其他存活的消费者。延迟队列必须采用这个模式。同时该模式不支持累计确认，即不支持批量确认

  • 灾备模式(Failover)：当存在多个消费者时，消息将按字典顺序排序，第一个消费者将作为唯一接收消息的消费者，如果该消费者断开连接，所有未被确认的消息和后续的消息将重新分配给队列中的下一个消费者。（只能对收到的最后一条消息进行取消确认）

  • 键共享模式(Key_Shared)：当存在多个消费者时，消息将根据其键进行分发，具有相同键的消息只会被分发到同一个消费者。这种模式下不支持批量消费，因为批量消费里面的数据的key不一定都归这个节点消费

消息预拉取

Consumer会默认预先拉取消息到Consumer本地，Broker侧会把这些已经推送到Consumer本地的消息记录为pendingAck，这些消息既不会再投递给别的消费者，也不会ack超时，除非当前Consumer被关闭，消息才会被重新投递。Broker侧有一个RedeliveryTracker接口，这个Tracker会记录消息到底被重新投递了多少次，每条消息推送给消费者时，会先从Tracker的哈希表中查询一下重投递的次数，和消息一并推送给消费者。

未确认的消息处理

unAckedMessageTracker中维护了一个时间轮，时间轮的刻度根据ackTimeout、tickDurationInMs这两个参数生成，每个刻度时间=ackTimeout/tickDurationInMs。新追踪的消息会放入最后一个刻度，每次调度都会移除队列头第一个刻度，并新增一个刻度放入队列尾，保证刻度总数不变。每次调度，队列头刻度里的消息将会被清理，unAckedMessageTracker会自动把这些消息做重投递。
重投递就是客户端发送一个redeliverUnacknowledgedMessages命令给Broker。每一条推送给消费者但是未ack的消息，在Broker侧都会有一个集合来记录（pengdingAck），这是用来避免重复投递的。触发重投递后，Broker会把对应的消息从这个集合里移除，然后这些消息就可以再次被消费了。

消息分块

https://blog.csdn.net/hefaji/article/details/136598177

  1. 如果producer发送的消息体大于配置的最大消息体，则消息就会被分块保存在broker
  2. 同一个大消息体的分块的，在broker中的分布式按照消息的顺序的，但是其中可能插入其他的消息体，不能统一保存完一个消息再保存下一个消息，会较大程度上影响producer的性能
  3. consumer在消费消息的时候，消费到最后一个的分块的时候，才会发送与该大消息关联的所有分块消息的确认，通知broker该消息体已接收到
  4. 分块对consumer的缓存和性能有一定影响，如上图，需要缓存两个message才能获取到所有的分块，如果大消息体多，则影响更强烈

plusar架构

整体架构

  • broker：负责处理producer发出的消息，并将消息给consumer消费；
  • BookKeeper（又称bookies）：bookies提供消息的持久化存储能力，broker将消息存储在bookies中。
  • ZooKeeper：ZooKeeper提供分布式配置和协调能力，存储归属信息、broker负载报告、bookies ledger信息等。

两层的架构，试的plusar可以更好的扩展（当然维护的难度也上去了）

  • Broker是无状态的，当需要更多的消费者或生产者时，可以简单地添加更多的Broker。触发负载均衡条件后，主题分区将在Brokers中做Load Balance，一些主题分区的所有权会转移到新的Broker。
  • Bookie是有状态的，当需要更多存储空间来将消息保存更长时间时，只需添加更多Bookie，流量将自动切换到新的 Bookie 中。Pulsar中不会涉及到不必要的数据搬迁，不会将旧数据从现有存储节点重新复制到新存储节点。

生产消费数据的流程：

Broker

  1. 所有的broker进行选举，选出leader进行主题的分配
  2. 将同一个namespace下的主题进行一只性hash，（如果是分区主题，则一致性hash的是partition），每个分区叫做Namespace Bundles
  3. 按照broker leader的分配，每个Namespace Bundles分配一个broker，即这里面的主题/分区后续将由这个broker接管处理，然后生产者和消费者与broker建立连接
  4. 如果后续发现，某个broker压力过大，或者broker宕机了，或者新增broker，则broker leader通过修改一致性hash上的分配节点修改对应的分配，然后生产者和消费者与新的broker建立连接
    • broker相关的信息都保存在zk上，无状态的broker不保存信息
    • broker性能负载压力等数据是通过CPU，内存等相关信息计算的，比较复杂的计算
    • 通过namespace上的配置，可以指定某个主题由哪个broker处理，但是这样broker就变成有状态的，不便于维护，同时对一致性hash的影响较大，可能导致整个集群的性能都下降，强烈不建议
    • broker的负载均衡，可以主动触发（定时任务扫描，比如新增一个broker进来），也可以被动触发（比如某个broker突然挂了，某个broker负载过高了）
