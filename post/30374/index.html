数据结构有哪些

大神笔记：https://www.cnblogs.com/xiaolincoding/p/15628854.html

映射关系如下：

string -  SDS
list   -  双向链表、压缩链表、listpack(未发布)
hash   -  压缩链表、哈希表、listpack(未发布)
set    -  哈希表
zset   -  压缩链表、跳表

SDS

  • 数据结构：字符串char[]、预申请空间长度、当前字符长度
  • 可以保存更复杂的数据结构，比如流，图片等编码文件
  • 避免内存重复申请，小于1M扩容时候double，大于1M扩容时1M
  • O(1)获取字符串长度，C语言获取长度复杂度O(n)

双向链表（linkedlist）

  • redis中的链表为双向链表，访问前后节点复杂度O(1)
  • 双向链表两个指针，占用空间大
  • 双向链表，元素分配散，对内存局部性和CPU缓存不友好

压缩链表（ziplist）

  • 类似数组结构，每个元素需要多大空间就分配多少空间，紧密排列在一起
  • 数据查询复杂度O(n)
  • 元素空间压缩，对内存局部性和CPU缓存及其友好
  • 不能保存过多元素，对查询速率有影响
  • 不能频繁增删元素，否则可能触发连续的内存变更操作，尤其是元素空间较大时
  • 大元素 & 长元素不适合用该结构

哈希表

  • 链式解决hash冲突
  • 渐进式refresh，query&update&delete在新旧hash表操作，add在新hash表操作
  • refresh时机：
    • 负载因子 > 1 && 没有RDB、AOF 触发
    • 负载因子 > 5 直接触发

跳表

  • 空间换时间的存储方式
  • 内部元素值需要有序
  • 用随机数的办法控制层数，保证每层数量差不多

quickList（暂时不考）

  • 也是双向链表，只不过之前的双向链表节点就是节点，这里面的节点是压缩链表
  • 治标不治本吧

mysql redis数据一致性

这类文章太多了，主要推荐：

核心：

引入缓存，提升系统的性能
保证高可用的情况下，让数据尽可能快速一致性。

方案

  • 不好的方案，而且都是显而易见的不好，而且数据过期之前可能一直是脏数据：

    • 先写mysql，再写redis
    • 先写redis，再写mysql
    • 先删除redis，再写mysql

  • 好的方案：先更新mysql，在删除缓存；

    • 这种情况下还是有可能导致数据异常的，当redis从DB捞出来数据之后，要往redis写的耗时，比mysql写进磁盘，然后删除redis存储的耗时还要高的时候，但是这种情况发生的概率就非常低了，因为写内存大概率比写磁盘快。
    • 万一删除缓存失败了，可以采用如下两种方案：
      • 引用kafka将删除任务异步化，进而达到最终一致性
      • 订阅mysql的binlog，进而达到数据最终一致性
    • 需要更加引用第三方的组件，增加的维护成本和系统的不确定性，但是①这样可以快速达到数据的一致性，且同时两个组件发生问题的概率不大，且都自带重试机制②引入缓存是为了加快系统的响应，且mysql底层由于隔离机制的问题，也可能导致数据的不一致性，如果要既要追求性能，也要保证实时一致，几乎是不可能的。③由于CAP理论，强一致性和高可用是无法同时满足的，强一致性可以通过数据库的Serializable，或者redis加锁，或者raft Paxos等实现，但是性能都不佳
