数据结构有哪些

大神笔记：https://www.cnblogs.com/xiaolincoding/p/15628854.html

映射关系如下：

string -  SDS
list   -  双向链表、压缩链表、listpack(未发布)
hash   -  压缩链表、哈希表、listpack(未发布)
set    -  哈希表
zset   -  压缩链表、跳表

SDS

  • 数据结构：字符串char[]、预申请空间长度、当前字符长度
  • 可以保存更复杂的数据结构，比如流，图片等编码文件
  • 避免内存重复申请，小于1M扩容时候double，大于1M扩容时1M
  • O(1)获取字符串长度，C语言获取长度复杂度O(n)

双向链表（linkedlist）

  • redis中的链表为双向链表，访问前后节点复杂度O(1)
  • 双向链表两个指针，占用空间大
  • 双向链表，元素分配散，对内存局部性和CPU缓存不友好

压缩链表（ziplist）

  • 类似数组结构，每个元素需要多大空间就分配多少空间，紧密排列在一起
  • 数据查询复杂度O(n)
  • 元素空间压缩，对内存局部性和CPU缓存及其友好
  • 不能保存过多元素，对查询速率有影响
  • 不能频繁增删元素，否则可能触发连续的内存变更操作，尤其是元素空间较大时
  • 大元素 & 长元素不适合用该结构

哈希表

  • 链式解决hash冲突
  • 渐进式refresh，query&update&delete在新旧hash表操作，add在新hash表操作
  • refresh时机：
    • 负载因子 > 1 && 没有RDB、AOF 触发
    • 负载因子 > 5 直接触发

跳表

  • 空间换时间的存储方式
  • 内部元素值需要有序
  • 用随机数的办法控制层数，保证每层数量差不多

quickList（暂时不考）

  • 也是双向链表，只不过之前的双向链表节点就是节点，这里面的节点是压缩链表
  • 治标不治本吧

RDB & AOF区别

主要参考：https://zhuanlan.zhihu.com/p/340082703

RDB

定义：基于配置，每间隔一段时间或者一些操作之后，把当前redis的快照压缩后写入磁盘

相关配置类似：

如果rdb过程中发生了写操作怎么办（COW(copy on write–和关注重构联系一下)）：

  • 数据在执行rdb的时候，是生成快照的，如果是读请求，则正常读即可，如果是写请求，则写请求会将当前数据拷贝到一块新的内存来做操作。故执行rdb的时候，保存的就是那一瞬间的快照数据

优势

  • 备份文件简单，只有一个文件，且文件较精简
  • 数据迁移和数据恢复比较快，直接拷贝这个文件。因为是数据快照，直接放到内存即可
  • 对性能影响较小，只有在特定时间才执行一次，其他时间无影响

劣势

  • 可能存在数据丢失，毕竟rdb的频次不是很高，数据丢失的时间期间可能比较长
  • 数据量较大时，rdb做数据备份会占用较长的时间，那一瞬间，对请求的读写也会有影响

AOF

定义：记录redis的所有写操作，保存在持久化文件中

优势

  • 基于上面不同的策略，可以选择实时性很高的持久化方式，基本上可以保证数据的完整性
  • 可以基于aof文件分析数据的操作过程等，可用于逻辑复现等

劣势

  • 备份文件较大，大于rdb
  • 备份文件恢复较慢，慢于rdb
  • 在redis写操作的时候，需要插入aof的执行，故对性能有一定影响

混合持久化

即是AOF + RDB一起做持久化，高版本默认开启

执行过程

基于类似下面的配置情况，会有一个叫做AOF重写的过程，在执行改操作的时候，就是混合持久化，当前的数据会做一次RDB快照，然后再把最新的数据用AOF记录起来。最后生成文件格式大致为[RDB file][AOF tail]

优劣势，就是双方的整合

  • 文件比RDB大，比aof小
  • rdb部分确定和rdb一样，可读性差。AOF部分文件可读性好，利用率不好，但是其实这部分数据量应该不大
  • 数据恢复的时候，因为前半部分采用了RDB，恢复得快。

redis单线程多线程

老版本都是单线程，新版本默认是多线程。（这里指的是命令的执行，RDB，aof等操作自然是由额外的线程处理的）

引入多线程的原因

  • 内存和CPU都是本地操作，耗时一般很低。但是客户端和服务端的网络的IO可能成为系统的瓶颈（IO多路复用）
  • 如果执行的命令是耗时操作，则可能导致redis相应变慢

多线程应用

  • 建立客户端与服务端链接的时候，每个线程处理一个链接，然后把建立成功的链接放在一个队列中，主线程（真正去执行命令的线程）从队列中取出数据执行。
  • 在执行耗时命令的时候，有对应的多线程命令用以处理。（执行过程中会加锁，避免其他的值过来修改）（del – unlink）(flushdb – flushdb async)(flushall – flushall async)

数据从内核态拷贝到用户态，需要经历两个流程

  • 询问内核态数据是否准备好了
  • 将数据从内核态拷贝到用户态

BIO

仅处理单个IO，数据从内核态拷贝到用户态持续阻塞，直到内核空间准备好了为止

NIO

仅处理单个IO，不断询问内核空间是否准备好，未准备好则返回错误，继续询问，直到内核空间准备好，才将数据从内核态拷贝到用户态

select

处理多个IO，向内核空间发起询问的时候，此时是阻塞住，内核空间会把当前select负责的所有都查看一遍，返回已经ready的IO。也是不断轮询。与NIO有点像，不过NIO仅负责一个，select负责一批

poll

与select基本相同，除了部分实现和结构不同，其他基本一致。见下表

redis快的原因

  • 存内存操作，速度快
  • 优质且定制化的数据结构优化
  • 主线程单线程执行，避免上下文切换，多线程竞争
  • 多路复用技术（BIO，NIO，select，poll，epoll），采用多线程较少客户端和服务端建立链接的时间gap，针对耗时操作，多线程处理

看门狗机制

执行一个任务的时候，可能会基于redis做一个加锁的操作，但是可能该任务还没有执行完，锁便到期了，这时候如果有其他线程处理可能就会有问题。

看门狗即是在这个锁还在的时候续期这个锁，等待主线程主动释放该锁

redis内存淘汰策略

内存满了的时候，根据配置执行不同的策略

（所有的key，有过期的key并不是维护全量的key，那样成本太高了，而是基于配置的数量N，仅维护N长度的可能会被淘汰的key）

  • Noeviction：默认策略，永不淘汰。除del等删除指令的写请求，直接return err
  • Allkeys_LRU：所有key，做最近最少使用（Least Recently Used）淘汰，基于使用顺序淘汰
  • Volatile_LRU：有过期时间的key，做最近最少使用（Least Recently Used）淘汰，基于使用顺序淘汰
  • Allkeys_LFU：所有key，做最近最不经常使用（Least Frequency Used）淘汰，基于使用次数淘汰
  • Volatile_LFU：有过期时间的key，做最近最不经常使用（Least Frequency Used）淘汰，基于使用次数淘汰
  • Allkeys_Random：所有key，随机淘汰
  • Volatile_Random：有过期时间的key，随机淘汰
  • Volatile_TTL：有过期时间的key，选择剩余时间最少得淘汰

过期的key怎么删除

目前redis默认采用的是惰性删除和定期删除的两种策略结合

  • 定时删除
    • 在写入有过期时间key的时候，创建一个定时任务用来删除key
    • 优点：内存友好，过期即可立刻删除
    • 缺点：CPU不友好，需要时刻监听删除任务事件
  • 惰性删除
    • 访问到数据的时候，如果发现该key已经过期了，则删除
    • 优点：CPU友好，只有在访问的时候才进行操作
    • 缺点：内存不友好，可能该key长时间不访问，则无法删除该key
  • 定期删除
    • 通过配置，每个1秒钟执行N次，随机捞取M个key，判断key是否过期，过期则删除
    • 优点：相对于定时删除，CPU友好，几乎不会额外占用高CPU资源，取决于配置。相对于惰性删除，可以删除长时间不访问的key
    • 缺点：需要手动调优，频率太高，则类似定时删除，频率太低，则类似惰性删除

一致性hash算法 && hash槽

一致性hash

参考：https://blog.csdn.net/weixin_43001336/article/details/122834046

分布式缓存中，如果使用普通的hash算法，（hashcode%节点数）来确定缓存数据在落在哪个节点，当节点数变化之后，整个缓存命中率就会快速下降，进而可能导致雪崩的问题

一致性hash指的是在hash之后，根据对应的值，找到下一个缓存节点

如果在集群中，增加删除节点，只会影响一部分数据，不会导致整个缓存雪崩

节点所在的位置，也是由节点的hashcode决定的，故无法保障节点是均匀部署在环上的，可能会导致部分节点缓存数据多，部分节点缓存数据少，故采用了虚拟节点的方式，同一个节点虚拟的存在在环上的不同位置

redis hash槽

参考：https://zhuanlan.zhihu.com/p/470368641

节点路由：

当客户端连接向某个master节点发送请求时，接收到命令的节点首先会通过 CRC-16(key)%16384 计算出当前key所属的slot，如果该slot由自己负责，直接处理响应客户端的请求，如果不是，则向客户端返回MOVED重定向请求，并将该slot对应的服务器节点的ip和port一并返回，客户端拿着这些数据重新访问。

节点增删：

节点增删的时候，只需要迁移部分数据即可，避免了内存雪崩的问题。但是迁移过程中，需要用户手动触发，迁移过程中可能导致数据不一致，故在迁移过程中尽可能避免写操作

手动迁移可见：https://blog.51cto.com/u_16213447/7028602

迁移过程中发生读写

参考：https://worktile.com/kb/ask/741982.html

  • 旧节点会继续处理来自客户端的读写操作，直到槽迁移完成。
  • 对于已经迁移的槽的读操作，旧节点会向新节点转发请求，并将新节点返回的结果直接返回给客户端。这样，客户端可以从旧节点获取已经迁移的数据。
  • 对于已经迁移的槽的写操作，旧节点会将写请求转发给新节点，并等待新节点返回写操作的结果。一旦新节点返回成功，旧节点也会返回成功给客户端。这样，客户端可以从旧节点获取写操作的结果。

优劣势

好处：

  • 比一致性hash实现起来比较方便。
  • 可以更方便得避免数据倾斜问题，不用采用虚拟节点就解决，可以根据服务器性能决定当前节点分配多少槽点
  • 同样可以避免缓存雪崩的问题

坏处：无法自动恢复，需要人工干预进行迁移。当前槽点是否可用主要依赖于主从节点的自主恢复

redis主从扩容

SYNC全量同步

  1. 新的从节点加入集群
  2. 从节点向主节点发送sync命令
  3. 主节点收到sync，开始执行rdb快照，并发送给从节点,并通过AOF的方式记录新增的写操作
  4. 从节点接收到快照，执行快照，生成数据
  5. 主节点向从节点发送aof文件，从节点执行aof文件。至此主从节点数据完全一致（因为这时候的aof文件只是新增的几个数据，文件比较小，且执行快，会阻塞主节点的线程）
  6. 后续主从按照正常的策略同步

PSYNC增量同步

  1. 如果是之前的从节点突然断开链接，然后突然连接上主节点，则会使用psync的同步策略。
  2. sync同步的是全量的数据，比较号性能，psync可能同步增量的数据，加快同步速度
  3. 在之前的同步过程中，主从节点分别记录喆数据的偏移量，且主节点会有缓存区：
    • 如果主从的偏移数据在缓存区内，则直接把缓存区的数据同步过去即可
    • 如果主从的偏移数据不在缓存区内，则执行上述sync策略，全量数据同步

同步策略

redis为了追求性能，主从之间的同步策略是异步的，不是强一致性，是最终一致性。
即redis主节点收到写命令的时候，在主节点执行完之后，会立刻返回给客户端，同时广播给从节点

解决：

  • 基于业务考虑，redis是做缓存的，不是存储的，如果是一致性要求不是很高可考虑忽略
  • 读写都只能通过主节点，从节点仅用于数据备份
  • 再引入一个cache，该cache强一致性，如果某个key发生过写操作，则记录在这个cache里面，下次读数据只能从主节点读取

PS:

  1. 一般主节点负责读写，从节点只负责读，如果客户端的写请求请求到从节点，从节点会将主节点的信息返回给客户端，让客户端再次发起请求
  2. redis的同步策略 && redis的故障恢复策略，大概率会导致数据丢失

哨兵模式

核心参考文章：https://blog.csdn.net/wzngzaixiaomantou/article/details/126089156

哨兵的作用

  • 监控（Monitoring）：哨兵会不断地检查主节点和从节点是否运作正常。
  • 自动故障转移（Automatic failover）：当主节点不能正常工作时，哨兵会开始自动故障转移操作，它会将失效主节点的其中一个从节点升级为新的主节点，并让其他从节点改为复制新的主节点。
  • 配置提供者（Configuration provider）：客户端在初始化时，通过连接哨兵来获得当前Redis服务的主节点地址。
  • 通知（Notification）：哨兵可以将故障转移的结果发送给客户端。

一主多从的redis集群和多主多从的redis集群，可以用同一套哨兵来监听

工作

哨兵本身是个集群，redis主从也是个集群，哨兵集群监听redis集群。

  1. 通过订阅的方式，发现哨兵之间建立联系，并获取到集群中的matser节点

  2. 哨兵通过INFO命令向主节点获取从节点列表，然后跟从节点建立链接，并保持PING来保证每个节点和哨兵集群之间的联系

  3. 当一个哨兵与master节点的联系断了，则会将当前master节点标记为主观下线，然后去咨询其他的哨兵，如果超过半数的哨兵认为master节点主观下线，则该master节点则为客观下线

  4. 此时认为集群异常，需要将从节点升级为主节点。哨兵之间通过raft算法进行master选举，用以redis主从数据库切换

  5. 选出新的master节点，规则如下：

    • 过滤列表中所有处于下线或断线状态的从服务器
    • 选择优先级最高的节点，每个节点的一个配置项
    • 如果优先级相同，选择offset最大的，offset表示主节点向从节点同步数据的偏移量，越大表示同步的数据越多；
    • 如果offset也相同，选择run id较小的；

  6. 其他节点根据新的master来做一次数据同步，之前的断开的master如果后续恢复了，作为从节点加入集群

redis为啥用跳表，innodb为啥用B+树

  • innodb用B+树
    • B+ tree是多叉树结构
    • 三层左右就可以存千万级别的数据量，跳表千万级别数据量不止3层
    • 一次查询，进行少数的IO便可做获取到数据
    • 查询性能较好
  • redis用跳表
    • 跳表是链表结构
    • B+树写入性能较差，需要旋转节点等操作
    • 跳表实现起来更加简单
    • 跳表写入性能较好
    • 读数据时不用考虑磁盘IO问题，且没有人会在redis缓存千万级别数据
