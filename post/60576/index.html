  • 阶段二：通过强弱三色不变式降低STW的时间

  • 阶段三：混合写屏障来避免STW的时间

    • 涉及堆空间：强弱三色不变式
    • 栈空间：引用到对象为全部黑色对象，而后创建的对象也为黑色对象

注意:只有在垃圾回收开始之后，混合写屏障的性能损耗

栈相关

栈空间回收：由栈自己控制，不由垃圾回收，但是垃圾回收会扫描栈

栈空间的自动管理

Go 的栈是动态可扩展的，每个 Goroutine 会有自己的栈，初始栈大小非常小（一般为 2KB 左右），但可以根据需要自动增长。栈的增长是按需进行的，当函数调用深度增加或者需要更多的栈空间时，Go 运行时会自动分配更大的栈空间。

栈的缩减

栈的缩减通常发生在栈空间不再需要那么大时。Go 运行时会监控 Goroutine 的栈使用情况，如果发现某个 Goroutine 在一段时间内的栈使用量显著减少，且之前扩展的栈空间变得不再需要，Go 运行时可能会将这些空闲的栈空间缩减回来。

栈空间与 GC

虽然栈空间本身不需要 GC 来管理，但栈上的变量可以引用堆上的对象。当栈上的变量生命周期结束时，这些引用可能会失效，GC 会识别到这些引用不再有效，并在下一次 GC 周期中回收这些堆上的对象。

主要涉及一下几个概念：mspan、mspanclass、mcache、mcentral、mheap

mspan：

8kb的golang页大小，为golang内存分配的最小单位

mspanclass：

一个以上的mspan组成数组，golang内存中有8kb~32kb的mspanclass，一共有67种，分别如下，为构成：

mcache：

GMP种每个P拥有一个mcache，根据需要向mcentral申请空间，申请的时候，需要加锁。之后的分配则无需加锁。包含两个mspan，分别用于存放指针对象和非指针对象，便于垃圾回收（GC不回收非指针对象）。

  • 比如当前有个17字节的对象需要，则需要去向mcentral申请空间>17的mspan链表，即class=3，申请之后，从mcache分配，如果是指针，则放到指针队列，如果非指针，则放到非指针队列

  • 微对象分配器：如果小于16字节且非指针，则为了提升空间利用率，用到这个分配器，就是8kb的空间，不间隔往里面放数据，比如同一个8位，前4位放入一个数，后4位放入一个数。正常的空间，8位如果没有放满，余下的空间就扔了，因为OS的读取都是8位的，余下的空间便于内存的读取

  • mcentral：mcache和mheap的中间商，mcache用完的空间还给mcentral，放在队列scav,mcentral空间不够了找mheap申请，放在队列free中。mcache申请的时候，优先从free申请，再从scav申请（因为这里面的空间可能被垃圾回收导致有余），再没有就向mheap申请

mheap：

提供给mcentral申请空间，上游对应的是操作系统的内存空间。如果程序运行过程中，有大于32kb的对象，也直接从这里申请，叫做arenas空间

综上：快的原因①预先申请了空间，避免频繁的申请空间，毕竟IO操作比较费时。②有比较好的内存管理制度，降级分配。

4、内存模型GMP(调度器)

参考大神文章：https://topgoer.com/%E5%B9%B6%E5%8F%91%E7%BC%96%E7%A8%8B/GMP%E5%8E%9F%E7%90%86%E4%B8%8E%E8%B0%83%E5%BA%A6.html

背景：

每个进程占用约4G内存，线程占用约4M，资源消耗都较大，故设计出了协程。 对CPU而言，还是在对进程内的线程进行运行，然后由线程对协程进行调度管理。 可以视为`进程 > 线程 > 协程`，上一级对下一级进行管理。

优势：

  • 协程轻量级的，一个协程只有几kb
  • 协程调度器运行起来方便，且可伸缩

老的调度器：

启用多个线程去全局协程队列里面抢占协程来执行

  • 创建、销毁、调度 G 都需要每个 M 获取锁，这就形成了激烈的锁竞争。
  • M 转移 G 会造成延迟和额外的系统负载。比如当 G 中包含创建新协程的时候，M 创建了 G’，为了继续执行 G，需要把 G’交给 M’执行，也造成了很差的局部性，因为 G’和 G 是相关的，最好放在 M 上执行，而不是其他 M’。
  • 系统调用 (CPU 在 M 之间的切换) 导致频繁的线程阻塞和取消阻塞操作增加了系统开销。

新的调度器

  • P的数量程序启动时定死，参数指定，默认是CPU核数，最大程度保障每个CPU都能被利用起来
  • M的数据>=P的数量，每个P都需要由一个M执行
  • 如果M被阻塞了，则将当前M和G挂起，待阻塞结束，G重回队列，然后获取一个M与当前P关联
  • M从本队列中取协程运行，不用加锁，减少竞争
  • M运行时创建的G默认放到本队列中，充分利用局部性原理。如果满了往全局队列放
  • 没有P关联的M不会立刻销毁，会等待几ms，避免线程的创建和销毁上的耗时
  • P为空时，则M从其他P中获取一般的G来运行，再没有则从全局队列中获取G运行
  • 所有的写成都被运行完之后，M和P则会进入自选状态，不会被销毁，保证下一个协程能快速运行

HashMap基础结构如下：

  • B决定了bmap的个数，进而map所能存储的数据大小
  • Key hash之后的后B为，决定了数据在哪个bmap中，头8位决定了在哪个
  • tophash保存键hash之后的头8位，先比较tophash是否相同，如果不同，则不用比较了，如果相同再去比较真正的Key是否相同，O(1)复杂度的原因
  • bmap中的数据保存结构为k/k/k/k/v/v/v/v，不是k/v/k/v/k/v/k/v,因为k一般比较小，这样可以更省空间
  • 用链表的方式解决hash冲突，指向下一个bmap(java采用的是数组解决，数组大了用红黑树)
  • map是协程不安全的，可以通过自己加锁或者sync.map来保证安全性

  • 下列每次操作都不加锁，所以非线程协程安全
  • GET:hash之后取根据后B位判断bmap，在根据前8位去bmap重查找，当前bmap无则去overflow中查找，若均无，则返回空
  • PUT:同GET，在GET过程中，记录下第一个空位，如果GET返回不为空，则更新，如果为空，则在第一个空位插入数据，如果无空位，则创建一个bmap作为溢出

扩容

等量扩容

随着不断的增删元素，会导致bmap的overflow很多，但是元素却不多，这个时候会把overflow的元素迁移到前面,删除多余的overflow，等价于紧缩map，桶的个数不会增加

时机：等量扩容也是把渐进式扩容，只不过新申请的空间和之前的空间是一样大小的noverflow >= uint16(1)<<B 同一个b里面溢出的链表长度是都大于2^B，扩容时使用sameSizeGrow做标记位

增量扩容

  • 负载因子为6.5（固定值），负载因子 = 元素总个数 / 桶的个数 ，即平均每个桶中元素个数 / 8

  • 渐进式扩容：增加元素的时候，判断负载因子 ，当需要扩容时，创建新的buckets，但是不会去初始化内存，只是预留，oldbuckets指向当前数据，然后只把当前值迁移到新的buckets，后续每操作一个数据迁移一个数据

    • 加快响应速度，避免因为全局桶的迁移，导致Map性能降低
    • 扩容过程中遇到如果发现还要扩容，则强制执行完当前的扩容，再进行下次扩容
    • B < 15，扩容为double，B >= 15，扩容为2^15

https://golang.design/go-questions/map/extend/ 一定要看

  • 等量扩容也是把渐进式扩容，只不过新申请的空间和之前的空间是一样大小的noverflow >= uint16(1)<<B 同一个b里面溢出的链表长度是都大于2^B
  • 扩容的时候是否会重新hash：等量扩容不重新hash，增量扩容需要重新hash，hash算法在src/runtime/alg.go中，如果CPU支持aes，则使用 aes hash，否则使用 memhash
  • map 只会扩容不会缩容，没有缩容的算法实现
  • 迁移过程是以b为单位进行迁移，不是以元素为单位进行迁移
  • 每个b都有自己的迭代器标识，iterator和oldIterator，读数据只会用其中之一，一旦b迁移完成，则会把oldIterator丢弃，后续也只会使用iterator
  • 迁移过程中，用nevacuate表示迁移进度，每次有一个b完成，则该字段+1
  • float 型可以作为 key，但是由于精度的问题，会导致一些诡异的问题，慎用之。

遍历为什么是无序的

  • 遍历的时候，假设发生了扩容，则元素的桶位置发生变化，则可能无序
  • go遍历的时候，生成了随机数，用于决定从哪里开始循环迭代。更具体的话就是根据随机数，选择一个桶位置作为起始点进行遍历迭代，因此每次重新for range map，见到的结果都是不一样的，比如2.4 和 2.4000000000000000000000001是相等的

扩缩容是否需要hash

golang map 扩容需要重新hash
mongodb 扩缩容不需要重新hash
主要取决于hash算法，golang的hash算法与分片数量有关（其实是库容的时候生成了一个随机数）
且golang的map的量级肯定跟mongodb的量级不能比，mongodb全部重新hash，得消耗多少性能

泛型（https://www.cnblogs.com/kevinwan/p/16223984.html）

使用场景：

  • 避免switch\case这种恶心的代码，或者reflect反射逻辑
  • 避免多个不同的方法因为参数不同重复写，比如比较大小

性能影响：

  • 虚拟方法表：运行时，根据T的类型，将类型赋予函数，再去执行。编译快运行慢
  • 单态化：根据T的类型，编译时增加每个方法的副本，运行时直接调用即可。编译慢运行快
  • GO：两种方法结合，如果可以推断出T的类型，比如上文中的Numeric，则采用单态化，否则使用虚拟方法表。虚拟方法表之前在GO中就被使用了。
  • 总结：性能影响不大，且不会成为瓶颈。考虑先优化函数实现，再考虑调用开销。

11、rpcx VS grpc

  • 1、 设计和背景

    • gRPC 是由 Google 开发的一个高性能、开源和通用的 RPC 框架。它支持多种编程语言，包括 C、Java、Go
    • rpcx 是 Go 语言生态圈的 RPC 框架。它利用 Go 语言的并发特性和简洁语法，实现高效的分布式 RPC 服务。

  • 2、 语言支持

    • gRPC 支持跨语言调用，目前支持的语言类型包括 C++、Java、Go、Python、Ruby、Node.js、Android、C#、PHP 和 Objective-C。
    • rpcx 主要使用 Go 语言开发，但也支持通过 HTTP 调用，使得其他编程语言也可以调用 rpcx 服务。

  • 3、 性能

    • gRPC 采用 ProtoBuf 序列化协议，性能优于其他协议，req\rsp包的大小相较于其他的小很多，而且包的encoding\decoding性能都相对较高
    • rpcx 在性能测试中表现优异，尤其是在高并发场景下，吞吐率和响应时间都优于许多其他框架。

  • 4、 服务治理

    • gRPC 目前不直接支持分布式服务治理，需要额外的开发来实现，不直接支持多种注册中心和管理中心。
    • rpcx 一样，支持服务治理，包括服务发现、负载均衡和容错机制、支持多种注册中心和管理中心。

  • 5、 序列化协议

    • gRPC 只支持 ProtoBuf 序列化协议。
    • rpcx 支持多种序列化框架，包括 Hessian2、Json 等，并且可以扩展。

Golang为啥不用尾递归优化：

  • 性能考虑，在很多场景中，非尾递归的性能差异影响不大，尤其是递归量不大时，几乎无差别
  • 垃圾回收，尾递归会增加垃圾回收机制的成本和性能，对于panic defer等函数的调用等也有影响
  • 可以采用等价的函数来进行实现尾递归，比如for循环或者显式栈来避免使用调用栈

13、Golang逃逸分析

逃逸分析是啥

  • 栈上分配：当变量的生命周期短且只在当前函数内使用时，编译器会将其分配在栈上。栈上的内存分配和释放非常快，因为它遵循后进先出的原则。
  • 堆上分配：当变量需要在函数返回后仍然存在（即超出函数作用域后仍然被引用），编译器会将其分配在堆上。堆上的内存由垃圾收集器（GC）负责回收，分配和回收的成本比栈高。

逃逸分析的过程

  • 引用传递：如果一个变量的引用被传递给其他函数，且这些函数在当前函数返回后仍然可能使用这个变量，那么这个变量会逃逸到堆。例如，返回指向局部变量的指针或在 Goroutine 中使用局部变量。
  • 闭包捕获：如果一个局部变量被闭包捕获，而这个闭包可能在当前函数返回后执行，那么这个变量会逃逸到堆。
  • 结构体的字段引用：如果局部变量是一个结构体，并且结构体的字段被传递到其他函数或存储在堆中，那么这个变量会逃逸到堆。

逃逸分析的好处

  • 内存管理优化：通过逃逸分析，编译器可以尽可能多地将变量分配在栈上，减少堆分配的次数，从而减轻垃圾收集器的负担，提高程序的性能。
  • 减少内存占用：栈上的内存分配是临时的，函数返回后会立即释放，这使得程序的内存占用更少。

14、协程为啥比线程快

https://python.itheima.com/news/20240517/103604.html

  • 低资源消耗: 协程空间一般只有几k，线程空间一般是几M
  • 轻量级调度: 协程在用户空间调度，线程在用户空间和内核空间的上下文切换（IO多路复用？）（有点像是零拷贝和IO多路复用，可以减少2次IO）
  • 无锁并发: 协程是运行在线程上的，共享同一个线程的地址空间，按照GMP模型，协程切换不用加锁的，但是线程切换，如果涉及到公共数据的访问，则需要加锁
  • 缓存的局部性: 协程是运行在线程上的，相关的数据缓存在线程中即可找到，但是线程是运行在进程上的，相关的缓存数据需要去进程中可能才能找到

线程切换相关：

线程切换是操作系统在多线程环境中进行任务调度的一种机制。以下是一些与线程切换相关的知识点：

  1. 上下文切换: 线程切换可能由多种因素触发，如当前线程执行完毕、发生中断、超时、高优先级线程变为可运行状态等，当操作系统从执行一个线程切换到另一个线程时，需要保存当前线程的状态（如寄存器、程序计数器等），并通过调度器加载新线程，通过一些内存映射或者加载的方式，获取新县线程上次运行停止的栈空间位置和数据，进而恢复新线程的运行。必须由操作系统内核空间操作，相对协程耗时

  2. 线程状态: 线程通常有几种状态，如就绪（Ready）、运行（Running）、阻塞（Blocked）和终止（Terminated）。

  3. 调度器 & 调度策略: 操作系统中的调度器负责决定哪个线程获得CPU时间，以及线程切换的时机,调度策略可以是先来先服务（FCFS）、轮转（Round-Robin）、优先级调度等

  4. 锁和同步: 在多线程程序中，使用锁（互斥锁、读写锁等）和其他同步机制(原子操作)来避免数据竞争和确保线程安全，需要避免死锁问题
