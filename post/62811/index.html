NOW直播-安全改造
2024-04-19 16:52:25

WHAT（安全指的是啥）

  • 底线类，不能涉及红线色情等明显违反国家的内容，对于存疑内容需要接入人审内容进一步审核
    • 用户个人资料(头像、昵称、签名、简介)
    • 用户言论动态(直播间弹幕，私聊文案，动态图文，评论信息)
    • 主播开播信息(主播间文案，主播间封面)
    • 直播内容审核(直播间画面，直播间言论)
  • 生态类，用户在平台不能出现过分行为，更多是基于场景配置，对应的数值需要业务评估调整
    • 上文底线类之后，其余的有平台评估(比如对平台的评论，不涉及底线，需要业务评估是否需要打击)
    • 用户刷榜数据(主要是针对商业化活动，比如用户通过某种行为冲榜的次数有最基础的频控)
    • 羊毛党用户(主要针对渠道拉新，比如用户来平台薅羊毛，不能全部拦截，也不能肆无忌惮)
    • IP设备频控(同一个IP、设备在同一个时间内操作次数管控)
    • 用户QQ微信端活跃度(针对用户在QQ、微信的行为进行评估是否允许操作)
  • 业务类，业务自己维护的能力：
    • 用户黑白名单(A拉黑B之后，则禁止某些行为)
    • 用户实名信息维护(未成年人禁止、送礼开播等行为)
    • 业务安全打击能力维护(封禁登录、开播、私信、房间管理员等行为)
    • 运营能力搭建(运营同学可以在平台上针对用户进行处罚解封)
    • 业务自身针对不同的场景维护的敏感词
    • 用户举报能力维护

WHY（为什么需要做这个事情）

  • 运营能力不足：甲方通报，平均一周能收到一个通报，主播为了利益，言论和画面上违规
  • 策略能力不足：多年前搭建的能力，对应的接入方已经不再维护，导致审核能力欠缺
  • 扩展能力不足：服务紊乱，之前每个服务对接不同的安全，接入的服务有几十个，统一维护难度大(核心技术难点)
  • 降本增效：梳理不用的安全送审，合并安全送审

HOW（怎么做的）

  • 策略升级：
    • 对于图片，语音，文字的审核能力，业务搭建成本大，接入第三方成熟的方案
    • 公司内部有多个不同的团队，分别负责底线，人审等，需要去多方了解沟通，评估接入的技术成本，运营成本，优劣势等，具体细节就不讲了，耗时了将近一个月，疯狂评估方案
    • 接入之后方案的切换，准确率，误伤率得保证
  • 扩展能力升级：
    • 百来个场景，优化后68种安全审核场景，整体链路缩减40%，服务节点减少60%，流量减少50%
    • 搭建统一的接入层，所有的安全送审都通过接入层接入安全，仅用5个服务cover安全送审（接入服务，数据提供服务，处置平台服务，业务安全服务，旁路送审）
    • 通过画布的拖拽决定使用哪些策略及其对应的配置阈值，新场景的接入目前几乎无开发量
    • 将对应的需要送审的场景逐一改造成新的送审，并逐步切换（工作量巨大）
    • 兼容同步送审和异步回调送审
  • 运营能力升级 & 降本增效
    • 舍弃无人维护的老平台运营，接入第三方的人审平台和公司内部的人审团队(之前承接给第三方)

结果：目前QQ直播、群视频采用的都是我这边的方案来做安全送审，且甲方的通报几乎降为0，非常偶尔来一个

降级 限流 熔断

case：

  1. 之前很多甲方通过的case，都是因为安全侧下游超时放过导致的，评估后决定从严处理，有异常直接拦截或者返回错误，让用户重试来解决
  2. 去年腾讯广州光纤断了&内部裁撤导致有个服务仅剩广州节点，导致用户进房发送弹幕都失败了
  3. 正常逻辑采用熔断去解决和降级，熔断一般作用在接口级别，意味着每个接口都需要加一下熔断逻辑，太复杂，且没必要
  4. 采用限流方式，限流反着用，错误数达到一直程度则放过，并将对应的请求写入kafka去重试，基于内部框架，直接配置化即可
  5. 设计了后续观察流量其实没有用上，这就是一个权衡的问题，需要具体业务具体分析。

降级

一句话，一切都是为了服务可能更好的进行下去，但是有些服务为了保证数据一致性是不能降级的，需要业务评估（比如结账，支付，个人觉得为了数据的一致性，在某些情况下直接返回异常）

  • 限流和熔断都算是降级的处理方式
  • 基础且必须字段过滤（uid，登录态，字段传参是否满足正则等）
  • 状态机过滤（顺丰）
  • 假设新A服务挂了，采用老的B服务作为备选
  • 传参某个字段异常，采用其他地方获取到的字段做备选
  • 同一个页面，分多个URL拉取数据，保证页面数据不会都丢失，也是一种降级策略
  • 在大型活动时，手动关闭部分不是核心数据的获取，也是降级策略

限流

参考：https://cloud.tencent.com/developer/article/2042810

作用：防止系统雪崩，避免给当前系统造成太大的压力；保证部分流量能够得到响应，超出系统承受能力的流量直接拒绝；一般作用在服务顶端

限流相关算法

  • 固定窗口
    • 使用计数器在周期内累加访问次数，当达到设定的限流值时，触发限流策略。下一个周期开始时，进行清零，重新计数。
    • 临界问题，两个周期的交界处，QPS可能会double
  • 滑动窗口
    • 将时间周期分为N个小周期，分别记录每个小周期内访问次数，并且根据时间滑动删除过期的小周期
    • 滑动窗口的格子划分的越多，那么滑动窗口的滚动就越平滑，限流的统计就会越精确。
  • 漏桶
    • 访问请求到达时直接放入漏桶，如当前容量已达到上限（限流值），则进行丢弃（触发限流策略）。漏桶以固定的速率进行释放访问请求（即请求通过），直到漏桶为空。
    • 请求量巨大时，也是稳定处理，QPS稳定
  • 令牌桶
    • 程序以r（r=时间周期/限流值）的速度向令牌桶中增加令牌，直到令牌桶满，请求到达时向令牌桶请求令牌，如获取到令牌则通过请求，否则触发限流策略
    • 可以短暂应对突发的流量，可以短暂应对高峰
算法 确定参数 空间复杂度 时间复杂度 限制突发流量 平滑限流 分布式环境下实现难度
固定窗口 计数周期T、周期内最大访问数N 低O(1）（记录周期内访问次数及周期开始时间） 低O(1) 否 否 低
滑动窗口 计数周期T、周期内最大访问数N 高O(N)（记录每个小周期中的访问数量） 中O(N) 是 相对实现。滑动窗口的格子划分的越多，那么滑动窗口的滚动就越平滑 中
漏桶 漏桶流出速度r、漏桶容量N 低O(1)（记录当前漏桶中容量） 高O(N) 是 是 高
令牌桶 令牌产生速度r、令牌桶容量N 低O(1)（记录当前令牌桶中令牌数） 高O(N) 是 是 高

设计模式

策略模型

  • 在合适的时候（服务启动），向map中注册方法及其对应的实现；在调用的时候，根据名字拉取到方法，然后执行调用
  • 安全中，这个用的最多，比如是否被拉黑，是否成年人，是否被频控，需要送审哪些安全第三方等方法，需要依赖不同的第三方，基于策略模式，就可以在合适的时候调用不同的方法策略

组合模型

  • 类似于树结构，一个完整的产品需要由不同的组件完成
  • 在一次安全校验的过程中，经常需要经过不同的策略，整合评估后才OK，这会就需要通过组合模型来并发去校验

工厂模式

  • 基于不同的传参，实例化不同的对象
  • 在框架中几乎用烂了，每个第三方组件都需要在服务配置中进行申明，调用的时候，需要根据名字实例化不同的对象，比如传入A_redis,则实例化ARedis，传入B_redis，则实例化BRedis

模板模型

  • 所有的处理都按照一定的模板进行执行，只有部分流程不同，不同的部分抽象成不同的方式来实现
  • 在安全处置的模块，这个模型用的特别多。针对处罚，按照模板（前置校验&参数补齐，PrePunish,Punish,PosPunish，数据入库推送等）新增一个类型的处置，仅需要实现接口，就可以完成接入，很多底层的逻辑都已经封装好了（比如前置过滤，上报，处置记录，推送等环节）

建造者模式

  • 在构建对象，需要不断地去填请求参数，这个时候就可以使用构造器模式
  • 在gorm中又大量的使用，比如db.Table("table").Select("field").Where("id = ?", 1).Where("index = ?", 2).Limit(1).Order("id ASC")
  • 在框架中这中模式也用了很多，比如proxy.DoRequest(ctx,req,opt...)后面的这些opt全是为了在这请求中，不断补充的参数。在构建框架层面的限流器的时候，也用了这种方式，很好用。
  • 但是除了框架层面用，业务代码中用，其实不见好用，因为可能很多opt都用不到

单例模式

https://www.liwenzhou.com/posts/Go/singleton/

  • 懒汉式，线程不安全，可能会在某一瞬间重复调用实例化方法
  • 懒汉式，线程安全，每次获取实例对象的时候都需要排队
  • 饿汉式（现在几乎都是用这种，简单，可能会造成实例化的对象根本用不到，且绝大部分是需要的，即使不需要，也不会占用太大的空间）
